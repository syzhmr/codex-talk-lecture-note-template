# 資料確認方針

このファイルは，手書きノート，板書写真，スライド，配布資料から情報を拾うための方針である。
このテンプレートでは，手書きノートを最優先資料として扱う。
ユーザーが `sources/` や `references/` 以下に入れた資料は，単なる保管物ではなく，原則として講演記録を作るための確認対象として積極的に使う。

## 資料優先順位

1. 手書きノート
2. 板書写真
3. スライド
4. 配布資料
5. 講義内で参照された参考文献

この順序は，講義中にユーザーが記録した情報を中心に整理するための優先順位である。
数学的な正確性，定理番号，記号，文献対応を確認する場合は，板書，スライド，配布資料，参考文献で照合する。
本文作成では，まず手書きノートに書かれている内容をすべて TeX 化する。
その後で，証明や紹介された主張・事実が参考文献のどこにあるかを補足する。

## 全転記の完了判定

手書きノートの各ページについて，少なくとも次を確認する。

- 節題，小見出し，番号付けを本文へ反映したか。
- 定義・定理・命題などの環境見出しを，手書きノートにある題名に従って反映したか。
- 定義している概念名や節題を，任意見出しへ重複させていないか。
- 人名または人名列による環境見出しを，括弧付きの帰属として反映したか。
- 文章，式，計算，定義，主張，定理，例，注意を本文へ起こしたか。
- 復習，今日の目標，前日からの接続，証明，証明の sketch，具体例を本文へ起こしたか。
- 証明，証明の sketch，曲線図，グラフ，可換図式，矢印，欄外メモ，丸囲み，下線を必要な形で本文へ起こしたか。
- 手書きノートの `proof` / `Proof` は `証明`，`proof sketch` / `sketch` / `証明の sketch` は `証明の sketch` として反映したか。
- 証明内の主張・補題・claim・lemma は，手書きノートが明示的に切り出している場合だけ本文でも切り出したか。
- 手書きノートで一続きの証明になっている部分を，本文側で新たな主張・補題へ分割していないか。
- 参考文献候補，URL，定理番号，ページ番号を台帳へ登録したか。
- 判読不能または未照合の箇所を `SOURCE_GAPS.md` に分離したか。

完成 PDF 本文には，この確認作業のページ番号や作業状態を書かない。
完成 PDF が元の手書きノートより大幅に短い場合は，全転記完了ではない可能性が高いため，ページごとの対応表を確認してから完成扱いにする。

## ユーザー投入資料の扱い

- ユーザーがフォルダに入れた資料は，まず `SOURCE_INDEX.md` または `references/REFERENCE_INDEX.md` に登録する。
- 登録済み資料は，対象話題ブロックに関係する限り，本文整理，引用確認，記号確認，資料間照合に積極的に使う。
- どの資料を使ったか，どの資料を確認できなかったかを作業報告に含める。
- 資料のファイル名だけで内容を推測せず，対象ページ，画像番号，スライド番号，節番号を確認する。
- 資料が多い場合でも，最初から無視せず，`SESSION_INDEX.md` と `SOURCE_INDEX.md` で優先順位と対応範囲を整理する。
- 権利，容量，判読不能，資料間衝突，講義記録からの距離に問題がある場合は，本文へ混ぜず `SOURCE_GAPS.md` に記録する。
- ユーザー投入資料に証明が含まれていても，講義中に記録されていない証明補完を本文へ追加しない。

## 手書きノートから抽出するもの

- 講義日，講義回，話題の区切り。
- 手書きノートの節題，小見出し，番号付け。
- 手書きノートの定義名，定理名，命題名，補題名，主張名。
- 定義，主張，命題，補題，定理，例，注意。
- 本文中の文章，式，計算，証明，証明の sketch。
- 講演者が強調したと思われる語句，丸囲み，下線。
- 記号，添字，曲線図，グラフ，可換図式，矢印，表。
- 証明の sketch として記録されているキーワード。
- 参考文献名，著者名，URL，arXiv 番号，DOI，Stacks tag。
- MathOverflow/Math StackExchange 番号。
- 定理番号，命題番号，補題番号，節番号，ページ番号。
- 色付き矢印，引き出し線，欄外メモが指す対応関係。
- 判読不能箇所と確認が必要な箇所。

手書きノートから拾った参考文献候補は，まず `HANDWRITTEN_REFERENCE_INDEX.md` に記録する。
手書きノートで英語表記の用語は，無理に日本語へ訳さず，手書きノートの表現として本文に残す。
PDF の目次や見出しに合わせるために，手書きノートの節題を勝手に言い換えない。
定義・定理・命題などの任意見出しも，手書きノートにない題名を補わない。
定義している概念名は，任意見出しではなく本文中の「...という」で処理する。
人名だけの定理名・命題名は，`定理 Mazur` ではなく `定理 (Mazur)` のように括弧付きにする。
`Prop A`，`Prop B` のような命題ラベルは，本文では `命題 A`，`命題 B` の形にする。
完成 PDF では，作業上のページ番号や「手書きノートでは」という言い方を本文に残さない。

## 板書写真から確認するもの

- 手書きノートの式や記号の判読。
- 講義中の話題順。
- 板書された定義，主張，定理名。
- 図式，交換図式，矢印の向き。
- 証明が板書された場合の見出しやキーワード。
- 講義中に書かれた参考文献名，URL，定理番号。

板書写真だけを根拠に，講義記録にない証明補完を行わない。

## スライドから確認するもの

- 講演者が用意した正式なタイトル，節構成，主張。
- 定理名，命題名，図，表。
- スライド番号，ページ番号。
- 参考文献欄，脚注，URL。
- 講義中の記法とスライド記法の差分。

スライドが講義中の実際の説明順と異なる場合は，手書きノートの順序を優先し，差分を `SOURCE_GAPS.md` に記録する。

## 配布資料から確認するもの

- 講義の正式な補助資料としての定義，仮定，主張。
- 定理番号，命題番号，ページ番号。
- 講義中に参照された参考文献。
- スライドや板書では省略された記号定義。

配布資料に証明があっても，講義中に記録されていない証明補完を本文へ追加しない。

## 色・記号の扱い

プロジェクトごとの色・記号の意味は `PROJECT_PROFILE.md` と `SOURCE_INDEX.md` に記録する。
判読できない場合は推測で本文へ混ぜず，`SOURCE_GAPS.md` に残す。

## 資料間衝突

次の衝突は `SOURCE_GAPS.md` または `NOTATION_CONFLICTS.md` に記録する。

- 手書きノートと板書で式が異なる。
- 手書きノートとスライドで主張の仮定が異なる。
- 配布資料と講義中の記号が異なる。
- 参考文献側の記法と講義中の記法が異なる。
- 定理番号，ページ番号，URL が資料間で食い違う。

衝突が解消するまで，本文では断定しない。
