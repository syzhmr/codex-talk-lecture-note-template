# 再現可能作業手順

このファイルを Codex 作業の入口にする。
目的は，毎回読む Markdown を減らしつつ，本文の品質判定を軽くしないことである。

## 最短チェックリスト

初見のモデルは，詳細節へ進む前に必ずこの順で確認する。

1. `git status --short --branch` を確認し，未コミット変更を把握する。
2. 依頼を「方針変更」「資料台帳更新」「手書きノート確認」「参考文献確認」「本文整理」「LaTeX 整形」「用語・記号確認」「Git 操作」に分類する。
3. `PROJECT_PROFILE.md` でプロジェクト固有情報，`GIT_WORKFLOW.md` で Git 運用を確認する。
4. 本文・数学内容に触る場合は `NOTE_POLICY.md` と `WORK_LEDGER.md` を読む。
5. `WORK_LEDGER.md` の「作業キュー」と「スモールステップ進捗」で対象ブロックと次の Step を固定する。
6. 手書きノート，板書写真，手書きメモは OCR にかけず，単ページ画像または元 PDF を目視する。
7. 手書き資料の見出し，式，図式，Recall，例，注意，証明，欄外注，レポート問題を圧縮しない。
8. 手書きにない写像名，説明的な主張文，証明補完，接続文を本文へ足さない。参考文献は補助照合に留める。
9. TeX 本文を変更したら `latexmk -g -dvi -interaction=nonstopmode main_final.tex` と `git diff --check` を実行する。
10. 作業を止める前に `WORK_LEDGER.md` へ更新内容，未解決点，次の Step を残す。

## 0. 作業開始

1. `git status --short --branch` を確認する。
2. ユーザーの依頼を次の型に分ける。
   - 方針変更
   - 資料台帳更新
   - 手書きノート確認
   - 参考文献確認
   - 本文整理
   - LaTeX 整形
   - 用語・記号確認
   - Git 操作
3. 型に応じて読むファイルを絞る。
4. 未コミット変更がある場合は，変更元と作業範囲を確認し，無関係な変更を壊さない。
5. ユーザーが `sources/` または `references/` に入れた資料は，未登録なら `WORK_LEDGER.md` または `references/REFERENCE_INDEX.md` に登録してから使う。

## 1. 常に読む

1. `WORKFLOW_STEPS.md`
2. `PROJECT_PROFILE.md`
3. `GIT_WORKFLOW.md`

## 2. 作業型ごとの読むファイル

| 作業型 | 読むファイル |
| --- | --- |
| 方針変更 | `README.md`, `PROJECT_PROFILE.md`, `NOTE_POLICY.md`, `WORKFLOW_STEPS.md`, `WORK_LEDGER.md`, `TERM_DICTIONARY.md`, `LATEX_SOURCE_POLICY.md`, `GIT_WORKFLOW.md`, `references/README.md` |
| 資料台帳更新 | `PROJECT_PROFILE.md`, `WORK_LEDGER.md`, `sources/README.md`, `references/README.md`, `references/REFERENCE_INDEX.md` |
| 手書きノート確認 | `NOTE_POLICY.md`, `WORK_LEDGER.md`, 対象の手書きノートまたは単ページ画像 |
| 参考文献確認 | `NOTE_POLICY.md`, `WORK_LEDGER.md`, `references/README.md`, `references/REFERENCE_INDEX.md`, 必要な `references/notes/` の個別メモ |
| 本文整理 | `NOTE_POLICY.md`, `WORK_LEDGER.md`, `TERM_DICTIONARY.md`, 対象 TeX |
| LaTeX 整形 | `LATEX_SOURCE_POLICY.md`, 対象 TeX。意味変更の可能性がある場合は `NOTE_POLICY.md` も読む。 |
| 用語・記号確認 | `TERM_DICTIONARY.md`, `NOTE_POLICY.md`, `WORK_LEDGER.md` |
| Git 操作 | `GIT_WORKFLOW.md` |

## 3. 本文整理の手順

1. 対象を，完成済み目次の最小単位または `WORK_LEDGER.md` の話題ブロックに固定する。
2. 対象ページ，対象資料，反映先 TeX，ロック済み範囲，未解決点を短く列挙する。
3. 手書きノートを OCR にかけず，1 ページずつ確認する。contact sheet だけを根拠にしない。
4. 見出し，小見出し，定義・確認/Recall・注意・例・定理・命題，本文，式，図式，矢印，欄外メモ，レポート問題，証明または sketch を抽出する。
5. 参考文献候補，記号衝突，判読不能箇所を `WORK_LEDGER.md` に分ける。
6. 本文へ反映する。手書きノートにない証明補完や接続文は入れない。
7. loss audit を行い，手書き資料にあった環境，図式，証明，脚注，引用，三点リーダー，アスタリスク標識が落ちていないか確認する。
8. 作業用表現が本文へ残っていないか確認する。
9. 本文を修正した場合は原則としてビルドする。
10. `WORK_LEDGER.md` の状態を更新し，完成扱いにできない理由を残す。

## 3A. 1 から完成版を作る手順

ユーザーが「1 から」「新規生成」「完成版を再現」と指示した場合は，通常の本文整理ではなくこの手順を使う。

1. 通常運用では，完成済み・ロック済み範囲を別枠で再生成しない。別枠生成は，ユーザーが明示的に実験または検証として指定した場合だけ行う。
2. 既存 `final/*.tex` を入力元として使わない。使える入力は，手書き資料，登録済み参考資料，台帳，辞書，プロジェクト方針に限る。
3. 対象ページごとに目標密度プロファイルを作る。見出し，環境，確認/Recall，例，注意，図式，脚注，引用，証明または sketch，欄外注，三点リーダー，アスタリスク標識を数え，`WORK_LEDGER.md` または作業メモに残す。
4. まず骨格だけを作る。日付，節，小節，小見出し，ページ境界，主張名，証明範囲を置く。この段階は完成候補ではなく，中間成果として扱う。
5. 次に全項目を転記する。例，注意，確認/Recall を地の文に吸収しない。証明または sketch は，定理・命題・補題・claim に対応する場合 `proof` 環境で囲む。部分証明や特定の場合だけの確認は optional title で範囲を明示する。
6. 図を文章だけに置換しない。可換図式，局所化図式，体の塔，曲線図，Hasse 図，元の所属記号が手書き資料にある場合は，TeX の図式として再現する。
7. 数式・記号を戻す。三点リーダー，アスタリスク標識，prime，hat/tilde，添字，肩字，場合分け条件，矢印ラベル，手書きにある写像名を，固定文字列検索と画像確認で照合する。手書きにない標準的な補助写像名は新設しない。
8. 参考文献対応を戻す。講義中に挙げられた文献は，台帳と本文引用のどちらで使うかを決める。参考文献が短く処理している箇所でも，講義中に展開された証明，図式，局所計算を削ったり，参考文献側の別証明へ置換したりしない。
9. loss audit を行う。目標密度プロファイルに対して，環境数，確認/Recall 数，proof 環境数，図数，引用数，脚注数，見出し数，三点リーダー，アスタリスク標識を比較し，減った項目を戻すか理由を記録する。圧縮が一箇所見つかった場合は，同じ講義回・同じ話題範囲で同型の圧縮を追加検索する。
10. 後続で仮定や記号が強くなった場合は，その話題だけでなく前段の目標，定理，記号導入へ戻して反映する。
11. ビルドし，`git diff --check` を通す。作業語や未確認語が本文にないことを `rg` で確認する。

1 から作った本文が短いこと自体は失敗ではない。
しかし，図，例，注意，引用，仮定，証明 sketch，三点リーダー，アスタリスク標識が失われた結果として短くなっている場合は完成扱いにしない。

## 4. 完成判定

完成扱いにするには，次を満たす。

1. 対象範囲の手書きノートをページ単位で確認している。
2. 本文が手書きノートより薄くなっていない。
3. 見出し，式，図式，欄外注，レポート問題の扱いが決まっている。
4. 参考文献対応，記号差，判読不能箇所が `WORK_LEDGER.md` または `references/notes/` に分離されている。
5. 本文に作業用表現が残っていない。
6. ロック済み範囲を勝手に変更していない。
7. 必要なビルドまたは検査を実行している。
8. 1 から作った範囲では，目標密度プロファイルと loss audit が残っている。

## 5. 停止条件

次の場合は本文へ推測で入れず，`WORK_LEDGER.md` に残す。

- 手書きノートが判読できない。
- 参考文献との仮定・対象・結論の差が解消できない。
- 記号衝突の採用記法を決められない。
- レポート問題や欄外注の対象範囲が画像上で確定できない。
- 証明補完が必要に見えるが，手書きノートに根拠がない。
- ユーザー確認でロック済みの範囲を変更する必要がある。

## 6. 報告

報告では次を短く書く。

1. 対象範囲。
2. 確認した資料。
3. 本文へ反映した内容。
4. 更新した台帳。
5. 残した未解決点。
6. 実行した検査またはビルド。
