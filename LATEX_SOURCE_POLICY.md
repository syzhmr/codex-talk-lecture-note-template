# LaTeX ソース可読性方針

このファイルは，LaTeX ソースを人間と AI の両方が読みやすい形に保つための補助方針である。
講演記録の内容，資料対応，参考文献引用，用語に関する判断は，必ず `WORKFLOW_STEPS.md`，`TALK_RECORD_POLICY.md`，`SOURCE_CAPTURE_POLICY.md`，`REFERENCE_CITATION_POLICY.md`，`TERM_DICTIONARY.md` を優先する。

## 優先順位

1. `WORKFLOW_STEPS.md` で作業型を確認する。
2. ユーザーの明示的な指示を最優先する。
3. 講演記録の内容については `TALK_RECORD_POLICY.md`，資料確認については `SOURCE_CAPTURE_POLICY.md`，引用については `REFERENCE_CITATION_POLICY.md` を優先する。
4. 既存のファイル構成，既存マクロ，既存のビルド方法を優先する。
5. この可読性方針は，意味を変えない整形・構造化・命名・差分管理のために使う。

## ファイル構成

- `main_final.tex` は文書全体の構造を管理する。
- 完成 PDF は表紙を置かず，小さめのタイトルと目次から始める。
- 完成 PDF の冒頭は，集中講義名，実施大学，時期，講演者，ノート，目次の順に置く。
- PDF 冒頭の人物情報は，講義・講演の形に合わせて `講演者: ...` とし，根拠なく `講師` に置き換えない。
- テンプレートには具体名を入れず，`preemble.tex` で設定する場合は `\title{<集中講義名>}` と `\author{<実施大学>\\<時期>\\講演者: <講演者>\\ノート: <ノート>}` のように複数行で書く。
- AI 利用を明記する場合は概要に置き，使用 AI の名称，モデル名，推論設定は実際の使用状況に合わせて断定的に書く。テンプレートには特定のモデル名を固定しない。
- 完成 PDF 本文には，作業用のページ参照，資料名主語，未確認ラベルを出さない。
- 文書クラスは article 型を基本にし，章見出しではなく概要と番号付き節で本文を進める。
- 目次の直後から本文を始め，大きな章扉や不要な改ページを置かない。
- 目次は `\notecompacttoc` と `\l@section` などの目次行設定で詰めて組み，本文の節番号や見出し構造を変えて目次だけを短くしない。
- 概要と日付つきの講義本文は本文としてひとつなぎにし，概要後の強制改ページは置かない。日付は直後の節見出しに近づけ，概要側の小見出しのように見えない余白にする。
- `preemble.tex` は既存のプリアンブルとして扱う。ファイル名は現状を尊重し，指示なく変更しない。
- 各日の本文冒頭の日付は，その日の最初のセクション見出しの直前に `\recorddate{YYYY年M月D日}` で表示し，目次には入れない。日付は太字にしない。
- 日付だけが前ページ末尾に残らないよう，`\recorddate` は直後のセクション見出しと同じページに置く。
- `final/talk01.tex` などの親ファイルは，話題ブロックへの `\input` を管理する。
- 各話題ブロックの本文は，`final/talk01/01-overview.tex` のように小さいファイルに置く。
- 完成 PDF 末尾の参考文献欄は `final/references.tex` に置く。

## ソース整形

- 本文は原則として 1 文 1 行にする。
- 段落途中で意味のない改行を大量に入れない。
- 長すぎる行は，文，式，条件，理由などのまとまりで分ける。
- 長い行を分割するときは，非空白文字列が変わらないようにする。
- `\href{...}{...}` や `\url{...}` の長い URL は，リンク先が変わる可能性があるため無理に途中で分割しない。
- 空行は，段落，大きな構造，意味の切れ目を示すために使う。
- 連続する空行は原則 1 行にまとめる。
- 環境，箇条書き，複数行数式は，既存の書き方に合わせて読みやすくインデントする。

## マクロとプリアンブル

- 既存マクロの意味を変更しない。
- 既存マクロと似た役割の新規マクロを不用意に増やさない。
- パッケージ読み込み順，LaTeX エンジン，文書クラスは指示なく変更しない。
- 使われていないパッケージの削除は，コンパイル確認と影響確認ができる場合だけ行う。
- 実プロジェクトからテンプレートへ `preemble.tex` や `main_final.tex` の運用を反映する場合は，汎用の定理環境，参照マクロ，ヘッダナビゲーション，ビルドに必要なパッケージだけを移す。分野固有の記号，文献名，人名，講義名，資料名，プロジェクト固有のマクロは持ち込まない。
- `\sourcepage`，`\noterefpage`，`\handref`，`\boardref`，`\slideref`，`\handoutref` などの作業資料対応マクロは，完成 PDF に作業資料名やページ参照を出さないため，テンプレートでは原則として空マクロにする。プロジェクトで明示的に必要な場合だけ，完成版に出す自然な出典注として別途設計する。
- ヘッダナビゲーションを使う場合は，`\notepagelinks` が存在するラベルだけを指すようにし，`main_final.tex` のビルドで未定義参照が残っていないことを確認する。
- Tate--Shafarevich 群の記号は `\Sha` を用い，`preemble.tex` の OT2/wncyr による定義を使う。本文でキリル文字を直接入力しない。

## ラベルと参照

- 既存ラベルは，不用意に変更しない。
- `\begin{dfn}[...]`，`\begin{thm}[...]` などの任意見出しは，手書きノートに題名がある場合だけ使う。
- 手書きノートで題名・小見出しとして書かれている語句は，本文側でも対応する見出しとして反映する。
- 見出しのためだけに手書きノートにない説明語を補わない。題名が手書きノート上の語句だけで書かれている場合は，その語句を基準に見出し化する。
- 定義している概念名，節題，説明用整理語は，定義・主張などの任意見出しにしない。
- 地の文でも，手書きノートにない分類語，説明語，評価語を補って本文の意味づけを変えない。
- 手書きノートにない接続語や導入文を足して本文を滑らかにしない。
- 定理名・公式名などを任意見出しとして残す場合は，`\begin{thm}[(公式名)]` のように括弧付きにする。
- 人名または人名列の任意見出しは，`\begin{thm}[(著者名)]` のように括弧を含めて書く。
- 内容整理のために作った説明語は，環境見出しにも本文の地の文にも入れない。必要な場合は作業台帳に置く。
- 新しいラベルを追加する場合は，節は `sec:`，定義は `dfn:`，定理は `thm:`，予想は `conj:`，補題は `lem:`，主張は `claim:`，命題は `prop:`，系は `cor:` を用いる。
- 話題ブロックには `block:` を用いる。
- 数式ラベルを追加する場合は `eq:` を用いる。
- 意味の薄いラベル名は避ける。
- 再参照しない補助的な記録には不要なラベルを付けない。
- 手書きページとの対応ラベルは，完成 PDF に出力しない内部コメントまたは作業台帳で管理する。

## 数式と証明

- 数式の意味，仮定，結論，番号の有無を変えない。
- 番号付き数式を番号なしに変更しない。番号なし数式を番号付きに変更しない。
- 講義中に記録された証明の sketch は整理してよい。
- 手書きノートが主張・補題として切り出していない証明内の文章を，本文側で新たな主張・補題として環境化しない。
- 講義中に記録されていない証明補完を本文へ追加しない。
- 参考文献に証明がある場合でも，講義記録の不足部分を補う目的で本文へ展開しない。
- 証明補完が必要に見える場合は，`SOURCE_GAPS.md` に記録する。

## 図表

- 図表の内容，順序，位置指定は指示なく変更しない。
- 図表を追加・差し替え・再作成する場合は，講義中に記録された情報として必要かを確認する。
- 板書写真の図式を LaTeX 図式へ起こす場合は，板書の矢印，対象，ラベルを確認し，推測で補わない。
- AI 生成画像，生成した画像，画像生成ツールで作った図，生成した模式図・背景・グラフは使用しない。
- 可換図式や代数的な矢印図は，`tikzcd` を最優先にして本文へ起こす。
- 原資料に可換図式がある場合は，文章説明だけで代替せず，本文内の近い位置に `tikzcd` 等で反映する。本文に入れない場合は，重複，判読不能，権利上の理由などを作業記録に残す。
- 可換図式の組版は，J. S. Milne, *The tikz package* / *Guide to Commutative Diagrams* (<https://www.jmilne.org/not/Mtikz.pdf>) を参照する。
- 通常の可換図式は `tikzcd` で書き，長いラベル，大きい対象，背の高いラベルがある場合は `column sep`，`row sep`，`swap`，`description` などで調整する。
- `tikzcd` で矢印の接続，曲線矢印，多重矢印，ラベル位置が読みにくい場合は，Milne のガイドに従い `tikzpicture` の `matrix of math nodes` を使う。さらに自由な配置が必要な場合だけ，直接 `\node` と `\path` で組む。
- 通常の同型関係は `X \cong Y` のように `\cong` を用いる。射の矢印の上に同型であることを表示する場合は `X \xrightarrow{\sim} Y` や `\arrow[r, "\sim"]` のように `\sim` を用い，矢印ラベルに `\cong` を置くのは原資料が明示する場合を除いて避ける。縦矢印で同型を示す場合は，横向きの `\sim` のまま置かず，例えば `\arrow[d,"f"',"{\rotatebox[origin=c]{90}{\ensuremath{\sim}}}"]` のように矢印方向に合わせて回転させる。
- チルダ系の `\tilde{}`，`\widetilde{}`，`\sim` とハット系の `\hat{}`，`\widehat{}` は TeX ソース上でも区別し，見た目だけで置換しない。
- 図式を編集したら，PDF または画像化したページで，矢印の向き，ラベル位置，対象の重なり，ページ幅からのはみ出し，式番号位置を確認する。
- 可換図式に元や所属記号が書かれている場合は，元の位置・所属記号・矢印を図式の一部として扱い，本文の地の文だけに吸収しない。矢印は対象ノードから伸ばし，元ラベルから伸ばさない。元や `\in` が矢印・ラベルに重なる場合は斜めまたは上下へ退避し，余裕がある箇所では無理に斜めにしない。`\in` は元ラベルの配置に合わせて回転し，完成 PDF の画像で元の位置，`\in` の向き，`Inf`/`Res` などの矢印横ラベルの位置・大文字小文字，同型記号の有無を目視確認する。
- 曲線図，グラフ，手書き形状そのものに意味がある図は，TikZ を最優先にして本文へ起こす。
- TikZ または `tikzcd` で正確に再現することがどうしても難しい場合だけ，手書きノート，板書写真，参考文献の該当箇所をトリミングして講義フォルダ内の `figures/` 以下に置き，`\includegraphics` で取り込むことを検討する。
- トリミング画像を使う場合は，著作権と共有範囲を考慮し，ユーザー作成ノート由来の画像を最優先にする。
- 参考文献 PDF，書籍，Web 資料など第三者資料のトリミング画像を本文に取り込む場合は，事前にユーザーの許可を取る。許可がない場合は取り込まず，TikZ/TikZ-cd で起こすか，本文で説明する。
- トリミング画像を使った場合は，元資料，ページ，トリミング範囲，使用理由，ユーザー許可の有無を `SOURCE_GAPS.md`，`references/notes/`，または作業記録に残す。
- 手書き画像を `\includegraphics` で入れる場合は，画像内に残る文章・式・ラベルが本文で転記・接続されているかを確認する。
- 画像内文字が本文と独立して読まれる状態を避ける。文字を TeX 化できる場合は TeX 化し，画像は図形だけに切る。
- 画像を取り込む場合も，完成 PDF 本文には「手書きノート p.X」のような作業用ページ参照を書かない。

## 引用・参考文献

- 引用キー，引用位置，参考文献情報を推測で変更しない。
- DOI，URL，出版年，著者名などの書誌情報は，信頼できる情報源で確認できる場合だけ補う。
- 本文で引用する前に `CLAIM_REFERENCE_MAP.md` で対応状態を確認する。
- 未確認の引用候補は本文へ出さず，`HANDWRITTEN_REFERENCE_INDEX.md` または `SOURCE_GAPS.md` に残す。
- 完成 PDF 末尾の参考文献一覧は，この記録で実際に使った文献だけを載せる。

## ビルド

- 既存の `latexmkrc` と現在のビルド方法を優先する。
- TeX 本文を変更した場合は，原則として `latexmk -g -dvi -interaction=nonstopmode main_final.tex` でビルドする。
- ビルド成功時に `main_final.pdf` と `main_final.synctex.gz` が生成されることを確認する。
- `main_final.synctex.gz` は SyncTeX 用の必要ファイルとして保持し，削除しない。
- 生成物や補助ファイルは，指示がない限り編集対象にしない。ただし `main_final.synctex.gz` は削除対象にしない。
- cleanup を行う場合も，`main_final.synctex.gz` を削除しない。削除してしまった場合は直ちに再ビルドして復元する。

## Git

このテンプレートを用いるプロジェクトでは Git 利用を必須とする。
ソース整形や本文整理の前に `git status --short --branch` を確認し，作業後に `git diff` と `git diff --check` を確認する。
内容変更と整形変更は commit を分ける。
手書きノート，板書写真，スライド，配布資料，参考文献 PDF，抽出画像は原則として Git に入れない。

## 報告

作業後は，次を簡潔に報告する。

- 変更したファイル。
- 主な変更内容。
- 実行したビルド，整形，検査コマンド。
- コンパイル結果。
- 未確認引用，記号衝突，資料間衝突。
- 意味を変えた可能性がある箇所があれば，必ず明示する。
